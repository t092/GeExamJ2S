# 表格與圖片處理建議

## 1. 現況分析

### 1.1 PDF 中圖表的實際型態

| 類型 | 實際格式 | 可擷取性 | 說明 |
|------|----------|----------|------|
| 地圖（等高線圖、行政區圖等） | **向量圖形**（SVG/路徑） | ❌ `get_images()` 無法擷取 | 以線條、填色繪製，非點陣圖 |
| 統計圖（長條圖、折線圖等） | **向量圖形** | ❌ 同上 | |
| 歷史照片、版畫、墓碑照片 | **嵌入點陣圖** | ✅ `get_images()` 可擷取 | 每份 PDF 約 8-15 張 |
| 示意圖、流程圖 | **向量圖形** | ❌ 同上 | |
| 資料表格 | **文字區塊** | ✅ PyMuPDF 抽取文字內容 | 但結構流失，變成碎片化文字行 |
| 圖表標籤（圖(一)、表(二)）| **文字區塊** | ✅ 可偵測文字 + 座標 | 已實現 mapping |

### 1.2 各年度統計

| 年度 | 總題數 | 含圖表引用題數 | 圖（圖） | 表（表） | 嵌入點陣圖 | 頁面數 |
|------|--------|---------------|----------|----------|-----------|--------|
| 112  | 54     | 25            | 16       | 9        | 8         | 15     |
| 113  | 54     | 29            | 24       | 6        | 51        | 15     |
| 114  | 54     | 27            | 21       | 6        | 15        | 15     |

## 2. 頁面渲染方案（已實現）

### 2.1 原理

用 PyMuPDF 將每個 PDF 頁面渲染為高解析度 PNG（200 DPI），完整保留所有向量圖形、地圖、表格的視覺呈現。

### 2.2 產出

```
assets/
├── 112/pages/page_01.png ~ page_15.png
├── 113/pages/page_01.png ~ page_15.png
└── 114/pages/page_01.png ~ page_15.png
```

### 2.3 圖表→頁面對應（已實現）

每題 JSON 已包含 `figure_pages` 欄位：

```json
{
  "number": 4,
  "stem": "圖(一)是某行政區內名為挖仔的聚落地圖...",
  "figures": ["圖(一)"],
  "figure_pages": { "圖(一)": 2 }
}
```

前端可根據 `figure_pages` 查找對應頁面渲染圖並顯示。

### 2.4 優缺點

| 優點 | 缺點 |
|------|------|
| 保真度 100%，所有圖表完整呈現 | 無法搜尋圖表中的文字 |
| 實作簡單，無需逐圖裁切 | 圖片檔較大（200 DPI 每頁約 2-3 MB） |
| 適用所有圖形類型（向量/點陣/混合） | 無法單獨顯示某一張圖（需裁切） |

## 3. 圖表區域裁切方案（建議實作）

### 3.1 原理

利用 PyMuPDF `page.get_text("dict")` 取得文字區塊座標，定位圖表標籤的位置（如 `圖(一)` 文字塊的 y 座標），往上推估圖表本體的邊界，從頁面渲染中裁切。

### 3.2 裁切邏輯

```
page_02.png (整頁)
    │
    ├─ 找到文字標籤：「圖(一)」在 y=566 的位置
    ├─ 往上找最近的題目文字結束位置 → 圖表頂部 (y=380)
    ├─ 左右邊界 → 頁面邊距 (x=45 ~ x=550)
    └─ 裁切出：crop(x=45, y=380, w=505, h=180) → fig_01.png
```

### 3.3 實作步驟

1. 讀取 `{year}_figures.json`（已含 label 座標）
2. 對每個 label，往上掃描文字區塊，找最接近的段落結束點作為圖表頂部
3. 從對應的 `page_NN.png` 裁切該區域
4. 輸出至 `assets/{year}/crops/{fig_id}.webp`

### 3.4 優缺點

| 優點 | 缺點 |
|------|------|
| 單張圖表檔案小，前端載入快 | 裁切邊界可能不精確（需人工校準） |
| 可直接嵌入題目旁顯示 | 不同年度版面可能有差異 |
| WebP 格式比 PNG 小 25-35% | 嵌入點陣圖與向量圖的裁切方式不同 |

## 4. 表格結構化方案（建議實作）

### 4.1 現狀

表格文字可被 PyMuPDF 抽取，但以碎片化行文字呈現，結構流失。

**原始 PDF 表格（視覺）**：
```
┌──────────┬──────┬──────┐
│ 電視臺    │ 正面  │ 負面  │
├──────────┼──────┼──────┤
│ 水果臺    │  28  │   4  │
│ F News   │   4  │   6  │
└──────────┴──────┴──────┘
```

**抽取後的文字**（碎片化）：
```
電視臺
占播報時間比例(%)
總比例正面報導負面報導
水果臺  36  28  4
F News  12   4  6
```

### 4.2 解析策略

```
Step 1: 偵測表格文字區塊
  └─ 利用文字座標，找出同一 y 範圍內多欄排列的文字

Step 2: 合併同列文字
  └─ 同一 y 範圍的文字行 → 合併為一列

Step 3: 推估欄位分隔
  └─ 以 x 座標分群，決定欄位邊界

Step 4: 輸出結構化資料
  └─ JSON 格式，含欄位名稱與資料
```

### 4.3 目標輸出格式

```json
{
  "number": 15,
  "tables": [
    {
      "label": "表(二)",
      "caption": "各電視臺報導時間比例統計",
      "headers": ["電視臺", "總比例", "正面報導", "負面報導"],
      "rows": [
        ["水果臺", 36, 28, 4],
        ["F News", 12, 4, 6],
        ["西水新聞", 7, 3, 2],
        ["W電視", 28, 4, 21]
      ],
      "footer": "統計時間：5/1 - 5/7 每日19:00~20:00"
    }
  ]
}
```

### 4.4 優缺點

| 優點 | 缺點 |
|------|------|
| 可在前端以 HTML table 重現 | 合併表格的解析較複雜（跨欄、跨列） |
| 資料可搜尋、可排序 | 部分表格座標混亂，需人工校正 |
| 結構化後可做資料題分析 | 不規則表格（如題組中的混合表格）難度較高 |

## 5. Vision AI 描述方案（opencode-see-image）

### 5.1 原理

使用 `opencode-see-image` 將頁面渲染圖送交 MiniMax M3 vision model，產出圖片/圖表的文字描述，儲存至題目 JSON。

### 5.2 流程

```
頁面渲染圖 (page_02.png)
  → opencode-see-image 工具
  → MiniMax M3 Vision
  → 輸出文字描述：
    "此圖為臺灣某行政區的等高線地形圖，顯示挖仔聚落
     位於河道轉彎處，周圍標高約50-100公尺..."
  → 存入 question["figures_description"]["圖(一)"]
```

### 5.3 環境設定

```bash
# 安裝（如已安裝則跳過）
opencode plugin opencode-see-image --global

# 或手動設定 config
# ~/.config/opencode/opencode.jsonc
{
  "plugin": ["opencode-see-image"]
}
```

預設使用 MiniMax M3（需 OpenCode Go 訂閱），或免費的 mimo-v2.5-free（需 OpenCode Zen）。

### 5.4 優缺點

| 優點 | 缺點 |
|------|------|
| 可產出圖表語意描述，利於搜尋與無障礙 | 需 API 成本（約 $2-5 USD 處理全部年份） |
| 可自動辨識圖表類型（地圖/統計圖/照片） | 描述品質非確定性，需人工抽檢 |
| 可輔助概念分類（如從地圖推斷考「臺灣地形」） | 部分複雜圖表可能描述不完整 |
